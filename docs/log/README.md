# 会话日志系统

**目的：** 在 AI 编码会话之间保持上下文，追踪架构决策、模式和经验教训。

---

## 为什么需要这个系统

**问题：** AI 在会话之间没有记忆。没有日志：
- 每次会话都从零开始
- 失败的方法会被重复尝试
- 架构决策被遗忘
- 上下文丢失

**解决方案：** 详细的会话日志作为：
- **AI 上下文** - Claude 读取这些来理解项目历史
- **开发者日志** - 你回顾什么有效/无效
- **架构编年史** - 追踪设计如何演变
- **决策记录** - 为什么我们选择特定方法

---

## 文件命名约定

```
YYYY-MM-DD-[sequence]-[short-description].md
```

**示例：**
- `2026-04-05-terrain-editor-setup.md`
- `2026-04-05-1-height-brush-fix.md`（当天第一个会话）
- `2026-04-05-2-vegetation-lod.md`（当天第二个会话）

**为什么：**
- 自动按时间排序
- 易于找到特定会话
- 一目了然看到做了什么

---

## 日志类型

### 1. 会话日志（主要类型）
**文件：** 使用 TEMPLATE.md 作为起点
**时机：** 每次与 Claude 的编码会话
**包含：**
- 我们尝试了什么
- 什么有效/无效
- 做出的决策
- 代码引用
- 下一步

**目标：** 未来的 Claude（或你）可以准确地从中断处继续

### 2. 决策记录（可选）
**文件：** `decisions/[decision-name].md`
**时机：** 经常被引用的重大架构决策
**包含：**
- 决策背景
- 考虑的选项
- 最终决定
- 权衡
- 理由

**示例：** `decisions/terrain-streaming-approach.md`

**目标：** 有一个单一的真相来源回答"为什么我们这样做？"

### 3. 学习文档（可选）
**文件：** `learnings/[topic].md`
**时机：** 发现值得记录的模式/陷阱
**包含：**
- 模式/技术描述
- 何时使用
- 常见错误
- 代码示例
- 相关会话

**示例：** `learnings/stride-shader-workflow.md`

**目标：** 建立项目特定的知识库

## 工作流程

### 开始会话

**1. Claude 读取（按顺序）：**
```
1. CLAUDE.md（规则和约束）
2. docs/ARCHITECTURE_OVERVIEW.md（当前状态）
3. docs/log/[最近会话].md（上次尝试内容）
4. docs/design/[相关文档].md（特定系统详情）
```

**2. 你简报 Claude：**
"今天我们要处理 [X]。阅读 [特定文档]。目标是 [Y]"

**3. 开始工作！**

### 会话期间

**边做边记录：**
- 复制 TEMPLATE.md → 新会话文件
- 工作时填写各部分
- 立即记录决策（记忆犹新）
- 记录失败的方法（为什么失败）
- 用 file:line 引用链接到代码

**不要担心完美** - 粗略的笔记比没有笔记好

### 结束会话

**收尾日志：**
1. ✅ 填写 "What Worked" / "What Didn't Work"
2. ✅ 记录任何做出的决策
3. ✅ 记录 "Next Session" 任务
4. ✅ 添加 "Quick Reference for Future Claude" 部分
5. ✅ 链接到变更的文件
6. ⚠️ 如需要更新设计文档
7. ⚠️ Git 提交并引用日志：`git commit -m "实现 X (见 log/2026-04-05-x.md)"`

### 下次会话

**Claude 从读取你的上一条日志开始** - 即时获得上下文！

## 记录什么

### ✅ 要记录这些

**决策：**
- 为什么我们选择方法 X 而不是 Y
- 我们接受的权衡
- 影响设计的约束

**失败的方法：**
- 我们尝试过但没用的方法
- **为什么失败**（根本原因）
- "不要再尝试这个，因为..."

**模式：**
- 发现的成功模式
- 要避免的反模式
- 可复用的解决方案

**解决的问题：**
- 症状 → 根本原因 → 解决方案
- 调查步骤
- 我们如何弄清楚的

**架构影响：**
- 设计变更
- 发现的新约束
- 需要的文档更新

**代码引用：**
- 关键实现的 `File.cs:LineStart-LineEnd`
- 链接到提交
- 引用测试

### ❌ 不要记录这些

**避免：**
- 复制粘贴整个文件（改为链接到它们）
- 每行代码的逐字记录
- 显而易见的东西（"导入了库 X"）
- 临时调试笔记
- 与代码无关的个人想法

## 日志结构哲学

### 核心部分（总是包含）

**1. Session Goal** - 我们要完成什么？
**2. What We Did** - 完成的工作
**3. Decisions Made** - 有意义的选择及其理由
**4. What Worked / Didn't Work** - 经验教训
**5. Next Session** - 从哪里继续

### 可选部分（相关时添加）

**Problems Encountered** - 如果遇到阻碍
**Architecture Impact** - 如果设计变更
**Code Quality Notes** - 性能、测试、技术债务
**Quick Reference for Future Claude** - 下次会话的上下文

### 指导原则

**要具体：**
- ❌ "修复了 bug"
- ✅ "通过添加锁修复了 TerrainSystem.cs:145 的竞态条件"

**解释为什么：**
- ❌ "改成 NativeArray"
- ✅ "为 Burst 兼容性改成 NativeArray（见性能指南）"

**链接一切：**
- 代码：`File.cs:Line`
- 文档：`[doc-name.md](../design/doc.md)`
- 之前的会话：`[session-name.md](./session.md)`

**为未来的你思考：**
- 3 个月后你能理解这个吗？
- Claude 能从中断处继续吗？
- "为什么"清楚吗？

- ✅ 阅读相关的历史会话
- ✅ 检查 `decisions/` 的相关选择
- ✅ 回顾 `learnings/` 的适用模式

## 与设计文档的整合

### 日志 vs 文档 - 不同目的

**会话日志（log/）：**
- 时间性 - "什么时候发生了什么"
- 探索性 - "我们尝试了 A、B、C"
- 上下文丰富 - "这是我们这样做的原因"
- 允许杂乱 - 粗略笔记

**设计文档（design/）：**
- 永恒性 - "这是它如何工作的"
- 规范性 - "这样做"
- 模式化 - "使用这种方法"
- 精炼的 - 真相来源

### 流程

```
1. 工作发生（记录在 log/）
2. 模式出现（多个会话）
3. 模式稳定（成为架构的一部分）
4. 更新 design/（正式化模式）
5. 未来会话引用文档（而不是日志）
```

**日志是原材料 → 文档是精炼产品**

### 何时更新文档

**更新 design/ 文档当：**
- 模式已成功使用 3 次以上
- 决策是永久的（非探索性）
- 实现与文档匹配（验证！）
- 其他开发者需要遵循这个模式

## 快速开始清单

**新会话：**
- [ ] 复制 TEMPLATE.md → `YYYY-MM-DD-name.md`
- [ ] 填写 Goal、Context
- [ ] Claude 读取上一会话 + 相关文档

**会话期间：**
- [ ] 决策发生时记录
- [ ] 记录失败方法（及原因）
- [ ] 链接到代码：`File.cs:Line`

**结束会话：**
- [ ] 填写 "What Worked" / "Didn't Work"
- [ ] 完成 "Next Session"
- [ ] 添加 "Quick Reference for Future Claude"
- [ ] 如需要更新设计文档
- [ ] Git 提交并引用日志

**下次会话：**
- [ ] Claude 首先读取上一条日志
- [ ] 检查 "Next Session" 的优先级
- [ ] 从中断处继续

## 成功指标

**好的日志 = 更好的 AI 会话：**
- ✅ Claude 问更少的"我们决定了什么？"问题
- ✅ 不重复失败的方法
- ✅ 新功能上手更快
- ✅ 清晰的架构演变轨迹
- ✅ 你可以在休息后继续工作

**如果日志没有帮助：**
- 太模糊？ → 添加更多具体内容
- 太长？ → 专注于决策，而不是逐字记录
- 不读它们？ → 让它们更易扫描
- 过时？ → 更新或删除
