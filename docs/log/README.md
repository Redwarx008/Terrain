# 会话日志系统

**目的：** 在 AI 编码会话之间保持上下文，追踪架构决策、模式和经验教训。

---

## 为什么需要这个系统

**问题：** AI 在会话之间没有记忆。没有日志：
- 每次会话都从零开始
- 失败的方法会被重复尝试
- 架构决策被遗忘
- 上下文丢失

**解决方案：** 详细的会话日志作为：
- **AI 上下文** — Claude 读取这些来理解项目历史
- **开发者日志** — 你回顾什么有效/无效
- **架构编年史** — 追踪设计如何演变
- **决策记录** — 为什么我们选择特定方法

---

## 会话启动流程（必须遵守）

### 新会话开始时，按顺序读取：

```
1. docs/ARCHITECTURE_OVERVIEW.md — 当前系统状态全貌
2. docs/CURRENT_FEATURES.md     — 功能完成度总览
3. docs/log/[最新会话].md        — 上次做了什么、从哪里继续
4. 根据任务读取相关文档：
   - 设计文档 → docs/design/
   - 架构决策 → docs/log/decisions/
   - 技术经验 → docs/log/learnings/
```

### 会话结束时，必须：

1. ✅ 创建会话日志到 `docs/log/YYYY/MM/DD/`
2. ✅ 如有重大架构决策 → 提取到 `docs/log/decisions/`
3. ✅ 如发现可复用模式或常见陷阱 → 记录到 `docs/log/learnings/`
4. ✅ 如系统状态变化 → 更新 `ARCHITECTURE_OVERVIEW.md` 和 `CURRENT_FEATURES.md`
5. ✅ Git 提交并引用日志路径

## 文件命名约定

```
YYYY-MM-DD-[sequence]-[short-description].md
```

**示例：**
- `2026-04-05-terrain-editor-setup.md`
- `2026-04-05-1-height-brush-fix.md`（当天第一个会话）
- `2026-04-05-2-vegetation-lod.md`（当天第二个会话）

**为什么：**
- 自动按时间排序
- 易于找到特定会话
- 一目了然看到做了什么

## 日志类型

### 1. 会话日志（主要类型）
**文件：** 使用 `TEMPLATE.md` 作为起点
**时机：** 每次与 Claude 的编码会话
**包含：**
- 我们尝试了什么
- 什么有效/无效
- 做出的决策
- 代码引用
- 下一步

**目标：** 未来的 Claude（或你）可以准确地从中断处继续

### 2. 决策记录（重大架构决策）
**文件：** `decisions/adr-[number]-[name].md`
**时机：** 经常被引用的重大架构决策
**包含：**
- 决策背景
- 考虑的选项
- 最终决定
- 权衡
- 理由

**何时提取：**
- 同一决策在 3 个以上会话中被引用
- 做出了不可逆的技术选择
- 影响多个子系统的设计决定

### 3. 学习文档（可复用模式和陷阱）
**文件：** `learnings/[topic].md`
**时机：** 发现值得记录的模式/陷阱
**包含：**
- 模式/技术描述
- 何时使用/不使用
- 常见错误
- 代码示例

**何时提取：**
- 同一模式被验证 3 次以上
- 同一失败被记录 2 次（"不要再尝试"）
- 框架特定的非显而易见行为

---

## 经验沉淀规则

- **同一决策被引用 3 次** → 提取为独立 ADR
- **同一模式被验证 3 次** → 提取到 `learnings/`
- **同一失败被记录 2 次** → 记录"不要再尝试"到 `learnings/` 的 Common Mistakes
- **每周回顾**：归档旧会话到 `archive/YYYY-MM/`，更新架构文档

## 维护

### 每周
- ✅ 回顾上周的会话
- ✅ 提取模式 → `learnings/`
- ✅ 提取重大决策 → `decisions/`
- ✅ 如需要更新设计文档

### 每月
- ✅ 归档旧会话到 `archive/YYYY-MM/`
- ✅ 更新 `ARCHITECTURE_OVERVIEW.md` 状态
- ✅ 更新 `CURRENT_FEATURES.md`
- ✅ 回顾 `learnings/` — 有什么要正式化到设计文档的？

### 开始新主要功能时
- ✅ 阅读相关的历史会话
- ✅ 检查 `decisions/` 的相关选择
- ✅ 回顾 `learnings/` 的适用模式

## 与设计文档的整合

### 日志 vs 文档 — 不同目的

**会话日志（log/）：**
- 时间性 — "什么时候发生了什么"
- 探索性 — "我们尝试了 A、B、C"
- 上下文丰富 — "这是我们这样做的原因"
- 允许杂乱 — 粗略笔记

**设计文档（design/）：**
- 永恒性 — "这是它如何工作的"
- 规范性 — "这样做"
- 模式化 — "使用这种方法"
- 精炼的 — 真相来源

### 流程

```
1. 工作发生（记录在 log/）
2. 模式出现（多个会话）
3. 模式稳定（成为架构的一部分）
4. 更新 design/（正式化模式）
5. 未来会话引用文档（而不是日志）
```

**日志是原材料 → 文档是精炼产品**
