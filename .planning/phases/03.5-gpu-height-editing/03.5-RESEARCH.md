# Phase 03.5: GPU Height Editing - Research

**Researched:** 2026-04-03
**Domain:** GPU compute shader height editing, CPU-GPU synchronization, multi-slice terrain support
**Confidence:** HIGH

## Summary

This research investigates implementing GPU-based height editing using compute shaders as an optimization over the current CPU-based approach. The existing implementation in `HeightEditor.cs` uses CPU iteration over brush radius pixels, modifying `ushort[] HeightDataCache`, then syncing to GPU via `Texture.SetData()`. This creates a CPU-GPU sync bottleneck for large brushes.

The codebase has a well-established compute shader architecture (`TerrainComputeDispatcher.cs`, `EditorTerrainComputeDispatcher.cs`) using Stride's `ComputeEffectShader` with `RWTexture2D<uint>` and `RWStructuredBuffer<T>` for GPU writes. The key challenge is that height textures are `R16_Float` (read-only shader resources), not UAV-capable, requiring architectural changes for GPU writes.

**Primary recommendation:** Create a dedicated GPU editing pipeline with UAV-capable height textures and a compute shader dispatcher that operates directly on GPU memory. Maintain a CPU-side undo/redo snapshot system that captures affected regions via GPU-to-CPU readback only when undo is requested (lazy readback), not during every stroke.

### Pattern 1: Compute Shader Height Editing
**What:** Dispatch compute shader to modify height values directly on GPU.
**When to use:** Large brush operations (>50px radius) where CPU iteration becomes bottleneck.
**Example:**
```hlsl
// Source: Adapted from TerrainBuildLodMap.sdsl pattern
namespace Terrain.Editor
{
    shader TerrainHeightEdit : ComputeShaderBase
    {
        // UAV-capable height texture (R16_Float with UnorderedAccess flag)
        stage RWTexture2D<float> HeightmapSlice;

        // Brush parameters passed via constant buffer
        cbuffer BrushParams
        {
            stage int2 BrushCenter;
            stage float BrushRadius;
            stage float BrushInnerRadius;
            stage float Strength;
            stage float FrameTime;
            stage int EditMode;  // 0=Raise, 1=Lower, 2=Smooth, 3=Flatten
            stage float TargetHeight;
        };

        float ComputeLinearFalloff(float distance)
        {
            if (distance <= BrushInnerRadius)
                return 1.0f;
            if (distance >= BrushRadius)
                return 0.0f;
            return 1.0f - (distance - BrushInnerRadius) / (BrushRadius - BrushInnerRadius);
        }

        override void Compute()
        {
            int2 texelCoord = int2(streams.DispatchThreadId.xy);

            // Boundary check
            int2 dimensions;
            HeightmapSlice.GetDimensions(dimensions.x, dimensions.y);
            if (texelCoord.x >= dimensions.x || texelCoord.y >= dimensions.y)
                return;

            float distance = length(float2(texelCoord) - float2(BrushCenter));
            if (distance > BrushRadius)
                return;

            float falloff = ComputeLinearFalloff(distance);
            float currentHeight = HeightmapSlice[texelCoord];
            float newHeight = currentHeight;

            // Edit modes
            if (EditMode == 0)  // Raise
            {
                newHeight = currentHeight + Strength * FrameTime * 0.015f * falloff;
            }
            else if (EditMode == 1)  // Lower
            {
                newHeight = currentHeight - Strength * FrameTime * 0.015f * falloff;
            }
            else if (EditMode == 2)  // Smooth (3x3 kernel)
            {
                float sum = 0.0f;
                int count = 0;
                for (int dy = -1; dy <= 1; dy++)
                {
                    for (int dx = -1; dx <= 1; dx++)
                    {
                        int2 neighborCoord = texelCoord + int2(dx, dy);
                        if (neighborCoord.x >= 0 && neighborCoord.x < dimensions.x &&
                            neighborCoord.y >= 0 && neighborCoord.y < dimensions.y)
                        {
                            sum += HeightmapSlice[neighborCoord];
                            count++;
                        }
                    }
                }
                float average = sum / float(count);
                float blendFactor = Strength * FrameTime * 0.075f * falloff;
                newHeight = currentHeight + (average - currentHeight) * blendFactor;
            }
            else if (EditMode == 3)  // Flatten
            {
                float blendFactor = Strength * FrameTime * 0.075f * falloff;
                newHeight = currentHeight + (TargetHeight - currentHeight) * blendFactor;
            }

            HeightmapSlice[texelCoord] = clamp(newHeight, 0.0f, 1.0f);
        }
    };
}
```

### Pattern 3: Compute Dispatcher Pattern
**What:** Encapsulate compute shader dispatch with parameter binding.
**When to use:** All compute shader dispatches following established pattern.
**Example:**
```csharp
// Source: Adapted from TerrainComputeDispatcher.cs:28-45
public sealed class HeightEditComputeDispatcher : IDisposable
{
    private const int ThreadCountX = 8;
    private const int ThreadCountY = 8;

    private ComputeEffectShader? heightEditEffect;

    public void Initialize(RenderContext renderContext)
    {
        heightEditEffect = new ComputeEffectShader(renderContext)
        {
            ShaderSourceName = "TerrainHeightEdit",
            ThreadNumbers = new Int3(ThreadCountX, ThreadCountY, 1),
        };
    }

    public void Dispatch(
        RenderDrawContext drawContext,
        Texture heightmapSlice,
        int brushCenterX, int brushCenterY,
        float brushRadius, float brushInnerRadius,
        float strength, float frameTime,
        int editMode, float targetHeight)
    {
        int threadGroupCountX = (heightmapSlice.Width + ThreadCountX - 1) / ThreadCountX;
        int threadGroupCountY = (heightmapSlice.Height + ThreadCountY - 1) / ThreadGroupCountY;

        var commandList = drawContext.CommandList;
        commandList.ResourceBarrierTransition(heightmapSlice, GraphicsResourceState.UnorderedAccess);

        heightEditEffect.ThreadGroupCounts = new Int3(threadGroupCountX, threadGroupCountY, 1);
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.HeightmapSlice, heightmapSlice);
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.BrushCenter, new Int2(brushCenterX, brushCenterY));
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.BrushRadius, brushRadius);
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.BrushInnerRadius, brushInnerRadius);
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.Strength, strength);
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.FrameTime, frameTime);
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.EditMode, editMode);
        heightEditEffect.Parameters.Set(TerrainHeightEditKeys.TargetHeight, targetHeight);

        heightEditEffect.Draw(drawContext);

        commandList.ResourceBarrierTransition(heightmapSlice, GraphicsResourceState.PixelShaderResource);
    }
}
```

### Anti-Patterns to Avoid
- **Using CPU HeightDataCache as source of truth with GPU editing:** Creates sync complexity. Instead, GPU texture becomes authoritative, CPU cache updated via readback only when needed.
- **Dispatching one compute shader per frame for continuous editing:** Causes GPU pipeline stalls. Batch multiple brush applications or use persistent UAV writes.
- **Creating new staging buffers each frame:** Causes allocation churn. Pre-allocate staging resources during initialization.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Thread group size calculation | Manual dispatch dimension math | `(width + ThreadCountX - 1) / ThreadCountX` pattern from TerrainComputeDispatcher | Off-by-one errors common |
| Resource barrier transitions | Skip barriers | `commandList.ResourceBarrierTransition()` | Required for UAV access on DX12 |
| Constant buffer layout | Manual byte packing | Stride shader parameter system (Keys class) | Auto-generated from SDSL |

**Key insight:** The codebase already has a robust compute shader infrastructure. GPU height editing is primarily about adapting the existing patterns, not inventing new ones.

## Common Pitfalls

### Pitfall 1: UAV Texture Format Mismatch
**What goes wrong:** `R16_UNorm` or `R16_Float` textures created without `TextureFlags.UnorderedAccess` cannot be bound as `RWTexture2D`.
**Why it happens:** Default texture creation only includes `ShaderResource` flag.
**How to avoid:** Always create editing heightmaps with `TextureFlags.ShaderResource | TextureFlags.UnorderedAccess`.
**Warning signs:** Compute shader fails to dispatch or silently does nothing; PIX/RenderDoc shows texture not bound as UAV.

### Pitfall 2: Missing Resource Barrier Transitions
**What goes wrong:** Texture corruption or GPU hang when reading UAV results without barrier.
**Why it happens:** DX12 requires explicit barriers between UAV and SRV states.
**How to avoid:** Follow the pattern in `TerrainComputeDispatcher.Dispatch()`: transition to `UnorderedAccess` before dispatch, then to `NonPixelShaderResource` or `PixelShaderResource` after.
**Warning signs:** Flickering terrain, incorrect heights, intermittent GPU device lost errors.

### Pitfall 3: Undo Snapshot Memory Explosion
**What goes wrong:** Storing full heightmap copies for undo history consumes excessive memory.
**Why it happens:** 8K heightmap = 128MB per snapshot; 50 undo levels = 6.4GB.
**How to avoid:** Use region-based snapshots (only store modified bounding box), or command-based inverse operations (RaiseBy X can be undone by LowerBy X).
**Warning signs:** OutOfMemoryException, editor slowdown after extended editing.

### Pitfall 4: Multi-Slice Boundary Artifacts
**What goes wrong:** Brush strokes crossing slice boundaries show seams or incorrect edits.
**Why it happens:** Each slice is a separate texture; compute shader only sees one slice at a time.
**How to avoid:** When brush center is near slice boundary, dispatch to both slices with clipped brush bounds; or use a unified staging texture for editing then copy back to slices.
**Warning signs:** Visible seams at slice boundaries, asymmetric brush results.

## Open Questions

1. **Undo/Redo Integration with GPU Editing**
   - What we know: Phase 4 (UNDO-01 to UNDO-05) is planned but not yet implemented. Current code has no undo system.
   - What's unclear: Whether undo should use CPU readback (slow but simple) or command-based inverse operations (fast but complex).
   - Recommendation: Implement command-based undo for GPU editing (store brush parameters, apply inverse on undo), fall back to CPU snapshot for complex operations like Smooth.

2. **Multi-Slice Editing Strategy**
   - What we know: `EditorTerrainEntity` supports up to 8 slices (`MaxBoundHeightSlices = 8`). Current CPU editing marks dirty slices and syncs each independently.
   - What's unclear: Should GPU editing dispatch to each affected slice separately (simple but multiple dispatches) or use a unified staging buffer (complex but single dispatch)?
   - Recommendation: Dispatch per-slice for simplicity; slice count is typically low (1-4 for most terrains).

3. **CPU-GPU Coherency Model**
   - What we know: Current architecture uses CPU cache as source of truth, syncing to GPU.
   - What's unclear: With GPU editing, should GPU become authoritative (CPU reads back when needed) or maintain dual sources with sync?
   - Recommendation: Make GPU authoritative during editing sessions. CPU cache updated via readback only when saving or for CPU-based operations (e.g., MinMaxErrorMap generation).

4. **Performance Threshold for GPU vs CPU**
   - What we know: CPU editing iterates `O(pi * radius^2)` pixels per frame; GPU dispatch is constant overhead + parallel execution.
   - What's unclear: At what brush radius does GPU become faster than CPU?
   - Recommendation: Benchmark both approaches. Tentative threshold: radius > 30px use GPU, otherwise CPU.
